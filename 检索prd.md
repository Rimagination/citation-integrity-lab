# PRD：AI 生成文本引文深度核查工具（极简/深度版）

## 1. 产品定位与目标

**产品定位**：一款面向科研人员、写作者和审稿人的轻量级 Web 工具。用户只需粘贴一段包含引用的文本（如一段综述或大模型生成的草稿）及参考文献列表，系统即可自动解析、高亮，并提供“真伪-相关性-支持度”三维度的深度体检。
**核心价值**：不在于“查得多”，而在于“查得透”。通过极简的单一文本框交互，提供最高质量的防幻觉、防移花接木的核查报告。

## 2. 核心场景与用户流程

**典型输入示例**：

> “在城市生态系统中，增加本地开花植物覆盖通常能够提高传粉昆虫多样性 [1]。此外，多项研究指出，湿地恢复后土壤有机碳库可在数年内逐步回升 (Garcia et al., 2021)。”
> [1] Lin, H., Zhao, R., & Chen, M. (2020). Native flowering plants increase pollinator diversity in urban green spaces. *Urban Ecosystems*, 23(4), 789-802.
> [2] Garcia, P., Miller, D., & Santos, L. (2021). Soil organic carbon recovery following wetland restoration: A global synthesis. *Ecological Applications*, 31(6), e02345.

**用户流程**：

1. **一键粘贴**：用户将上述整段文本（正文+尾部参考文献）直接粘贴进网页中央的巨大输入框。
2. **智能解析**：点击“深度核查”后，系统瞬间将普通文本转化为富文本。正文中的 `[1]` 或 `(Smith et al., 2023)` 变成带有状态颜色（🟢 🟡 🔴）的可点击标签。
3. **沉浸式复核**：用户点击标红的标签，右侧/悬浮窗立刻弹出诊断面板，直指核心问题（例如：“元数据错配：该 DOI 对应的真实论文并非关于湿地恢复碳库”，“断言不支持：摘要中未提及传粉昆虫多样性提升”）。

## 3. 核心功能需求 (Functional Requirements)

### 3.1 智能文本解析 (Smart Parsing)

由于只有一个输入框，系统必须具备极强的文本结构化能力。

* **FR-1.1 正文与引文分离**：通过轻量级 LLM Prompt 或启发式正则，自动切分输入框中的“正文断言部分”与“参考文献列表部分”。
* **FR-1.2 引用锚点识别**：精准提取正文中的引用标记（支持数字 `[1]`、作者-年份 `(Author, Year)`），并与尾部的参考文献条目建立 1 对 1 或 1 对多的映射关联。

### 3.2 三维深度核验引擎 (3D Verification Engine)

系统对解析出的每一对“正文引用处 ↔ 参考文献”执行三层递进核查：

#### 维度一：身份真伪与元数据对齐 (Authenticity & Integrity)

* **FR-2.1 双源验证**：提取条目中的 DOI、标题或作者，并发请求至 Crossref 和 OpenAlex 开放元数据接口。
* **FR-2.2 幻觉拦截**：
* 检测“纯捏造”：查无此文。
* 检测“移花接木”：用户提供的标题/作者，与通过该 DOI 查到的官方元数据进行 Diff 对比。若核心字段（如标题、年份）相似度低于 80%，打出 🔴 `核心元数据冲突` 标签。



#### 维度二：语境与意图匹配 (Contextual Relevance)

* **FR-3.1 摘要获取**：通过 OpenAlex 获取被引文献的官方 Abstract。
* **FR-3.2 语义相似度与意图**：
* 提取“正文引用所在句 + 前后句”作为 Citation Context。
* 使用轻量级嵌入模型（如 SciBERT）计算 Context 与被引文献摘要的向量相似度。
* 使用 SciCite 意图分类模型，判断此处引用是“背景”、“方法”还是“结果”。若正文在谈具体参数，但引用的却是一篇宏观综述，打出 🟡 `引用意图偏离` 标签。



#### 维度三：断言支持度 (Claim Support / NLI)

这是本产品的“杀手锏”功能。

* **FR-4.1 断言抽取**：将正文引用所在句提取为一个个独立的 Claim（例如：*Claim 1: 湿地恢复可在数年内显著提升土壤有机碳库*）。
* **FR-4.2 自然语言推理 (NLI)**：使用大模型判定被引文献的摘要/结论是否支持该 Claim。
* **FR-4.3 结论输出**：明确给出 `🟢 完全支持`、`🟡 部分支持/仅相关`、`🔴 不支持/矛盾`、`⚪ 证据不足（仅凭摘要无法判断）` 四种结论，并附带判断理由。

## 4. 界面与交互设计 (UI/UX)

* **初始状态**：极简的居中大文本框（类似 DeepL 的界面感），带一个“开始核查”主按钮。
* **核查中**：文本框内的文字开始流式渲染，识别出的引用标记变成加载状态的 Spinner。
* **结果状态（所见即所得视图）**：
* **左侧工作区**：保留用户的原始排版，但所有引用标记高亮显示（红黄绿三色表示健康度）。
* **右侧/侧边栏诊断卡片**：点击任意高亮标记，右侧展示详情：
1. **官方真实卡片**：展示从开放库拉取的真实文献信息（提供一键复制正确 BibTeX/纯文本的功能）。
2. **冲突对比区**：红字高亮显示用户文本与真实信息的差异。
3. **支持度雷达**：一句话解释“为什么这篇文章不支持你的正文”。





## 5. 系统架构建议 (简化版)

* **前端**：React + 类似 ProseMirror 的富文本编辑器内核（用于实现精准的文本锚点高亮和点击事件）。
* **后端 (API 驱动)**：
* `POST /api/parse`：基于 LLM 的文本分离与映射。
* `POST /api/verify/metadata`：请求 Crossref/OpenAlex。
* `POST /api/verify/support`：调用云端 LLM（或本地部署的 RAG 判断模型）进行 Claim 抽取和支持度推理。
*由于去掉了批量 200 篇和复杂文档支持，系统可采用更直接的同步请求或轻量级 Server-Sent Events (SSE) 向前端推送单条核查进度。*
